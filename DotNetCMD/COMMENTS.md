# DotNetCommander — архітектурні нотатки

> Цей файл пояснює технічні рішення й межі відповідальності компонентів.
> Фактичний стан коду зафіксовано в `AGENTS.md`, майбутні роботи — у `ROADMAP.md`, виконані зміни — у `CHANGE.md`.

## Загальний напрямок

DotNetCommander лишається Windows-first застосунком на WinForms. Для двопанельного файлового менеджера це практичний вибір: `ListView` добре працює з великими каталогами, застосунок швидко запускається, а Win32 і shell API доступні без додаткового UI-шару.

Проєкт не намагається буквально копіювати Total Commander або Double Commander. Його власний профіль формують `Quick View`, внутрішні viewer/editor форми, порівняння файлів, асинхронні операції та commander-style керування з клавіатури.

## Межі відповідальності

- `AppForm` координує дві панелі, меню, панель дисків, F-кнопки й глобальні команди. Файлову та предметну логіку не слід повертати у форму.
- Commander-style командний рядок належить головному layout `AppForm`: форма керує фокусом, історією та вставленням назви поточного елемента, а запуск введеної команди делегує Windows-специфічному `WinCommandLine`.
- `FileBrowser` відповідає за представлення каталогу, навігацію, виділення, сортування та взаємодію конкретної панелі з користувачем.
- `CommandService` є точкою виконання основних команд `view/edit/copy/move/delete/new file`.
- `FileOperationService` планує і виконує `copy/move/delete`; форми операцій показують стан і збирають рішення користувача.
- `FileTypeClassifier` є єдиним власником визначення archive format за розширенням і сигнатурою; `ArchiveService` лише виконує pack/unpack через `System.IO.Compression` і `System.Formats.Tar`, а UI не читає archive entries самостійно.
- `BrowserPanelBase` задає спільний контракт location/items/selection/capabilities/navigation/refresh для фізичної та віртуальних панелей.
- `FileBrowser` є сумісним host для наявного `AppForm`, а `ArchiveBrowser` — read-only реалізацією віртуальної панелі; така межа дозволяє додавати інші провайдери без копіювання UI-контракту.
- `FileSystemService` централізує доступ до файлової системи там, де вже проведено цей рефакторинг.
- `QuickViewControl` інкапсулює preview і спирається на спільну класифікацію типів файлів.
- `FileTypeClassifier` узгоджує визначення типів між Quick View, окремими viewer формами та порівнянням файлів.
- `LogService` є спільним фасадом логування; критичні сценарії не повинні створювати паралельні механізми журналювання.

## UI та діалоги

`Dialogs/Common/` містить загальні commander-style діалоги, а `Dialogs/Operations/` — форми довгих файлових та архівних операцій. Спільна типографіка проходить через `DialogStyleService`, а користувацькі параметри — через `Properties.Settings` і `SettingsStorage`.

`Dialogs/Legacy/` — історичний шар зі збереженим legacy API. `InputBox`, `frmOptions` і `frmWait` уже використовують типографіку та візуальні принципи актуальних форм, але цей каталог не задає архітектурний шаблон для нових діалогів.

Редактори живуть у `Editors/`, а окремі переглядачі — у `Viewers/`; ресурси та вузькоспеціальні залежності зберігаються поруч із відповідним editor/viewer. Їхні форми мають залишатися узгодженими за базовими діями: закриття, збереження, `Save As...`, status bar і `F1`-довідка там, де ці дії доречні.

## Платформні межі

- Практичний фокус — сучасний Windows і `net8.0-windows10.0.22621.0`.
- `Infrastructure/Platform/OS.cs` лишається фасадом платформної логіки, а Windows- і Unix-реалізації згруповані в цій самій гілці.
- Windows shell-сценарії мають окремо враховувати файли, каталоги, `.lnk`, асоціації, `.bat/.cmd/.ps1` і недоступні мережеві ресурси.
- `WinCommandLine` запускає введений текст через системний command processor у фізичному каталозі активної панелі; `Enter` завершує консоль після команди, а `Shift+Enter` використовує persistent-режим.
- `.lnk` обробляються без обов'язкової `x86-only` interop-залежності.

## Правила для подальших змін

- Нові файлові операції розвивати через `FileOperationService`, а не через event handlers форм.
- Розпізнавання архівних форматів розвивати через `FileTypeClassifier`, а їх виконання — через `ArchiveService` з обов'язковою перевіркою extraction paths і безпечними тимчасовими файлами.
- Майбутні FTP, search-result або інші virtual panels мають наслідувати `BrowserPanelBase` і декларувати можливості через `BrowserPanelCapabilities`.
- Shell-іконки отримуються через `FileIconCache`: ключ базується на типі (каталог, файл без розширення або нормалізоване розширення) та розмірі іконки, тому нові panel providers не повинні напряму опитувати OS для кожного елемента.
- Спільний доступ до шляхів і файлової системи проводити через сервісний шар, щоб системні edge cases мали одну точку обробки.
- Нові типи preview/view/compare узгоджувати через `FileTypeClassifier`.
- Нові UI-рядки додавати через `Language.cs` і `Resources/Language*.resx`, з робочим англійським fallback.
- Зміни гарячих клавіш перевіряти разом у `AppForm`, `FileBrowser`, меню та нижній F-панелі.
- Конкретні незавершені завдання не додавати сюди: вони належать `ROADMAP.md`.
