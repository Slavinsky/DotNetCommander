# DotNetCommander Roadmap

> Цей файл містить лише незавершені напрями розвитку.
> Поточний стан коду описано в `AGENTS.md`, історію виконаних змін — у `CHANGE.md`.

## Мета

Поточний вектор розвитку після версії `1.6.0`:

1. Довести файлові операції до рівня щоденного використання без сюрпризів.
2. Закрити залишки платформних і shell-інтеграційних прогалин.
3. Підготувати кодову базу до подальшого розширення без накопичення технічного боргу.

## Принципи roadmap

- Не дублювати Windows Explorer, а підсилювати commander-потік роботи.
- Нові можливості додавати лише після стабілізації базових сценаріїв `copy/move/delete/view/edit`.
- Логіку по можливості виносити з форм у сервіси та допоміжні класи.
- Кожну нову функцію відразу перевіряти на локалізацію, великі каталоги та гарячі клавіші.

## Етап 1 (v1.7). Harden File Operations

Ціль: зробити існуючі файлові операції передбачуваними в конфліктних і "брудних" сценаріях.

- Розширити вже додані діалоги конфліктів підсумками операцій, retry/recovery сценаріями і кращою звітністю.
- Коректно обробляти часткові збої під час пакетних `copy/move/delete`.
- Явно показувати підсумок операції: скільки елементів завершено, пропущено, з помилками.
- Підготувати основу для черги операцій без блокування UI.
- Перевірити поведінку на великих деревах каталогів і змішаних наборах файлів/папок.

Очікуваний результат:

- `FileOperationService`, `FormCopy`, `FormDelete` закривають більшість реальних користувацьких сценаріїв без "тихих" збоїв.

## Етап 2 (v1.7). Error Handling and Diagnostics

Ціль: спростити діагностику проблем і зменшити кількість непрозорих падінь.

- Уніфікувати використання `LogService` у всіх критичних сценаріях.
- Уніфікувати повідомлення про помилки для файлових операцій, preview та редакторів.
- Визначити, де достатньо `MessageBox`, а де потрібен структурований звіт про помилку.
- За потреби розширити `PerfTrace` для вимірювання повільних сценаріїв, лишивши його вимкненим за замовчуванням.

Очікуваний результат:

- Критичні проблеми відтворюються за логами, а користувач бачить зрозуміле повідомлення замість "щось не так".

## Етап 3 (v1.8). Platform Robustness

Ціль: прибрати системні обмеження, які заважають стабільній роботі на сучасному Windows.

- Додати підтримку довгих шляхів і перевірити сценарії з глибокими деревами.
- Пройтися по Unicode-іменах, системних атрибутах, read-only файлах, hidden/system каталогах.
- Окремо перевірити `symlink`, `.lnk`, недоступні каталоги та мережеві шляхи.
- Вирівняти контракт між `OS.cs`, `Win32.cs` і `Unix.cs`, якщо кросплатформений шар лишається частиною проєкту.

Очікуваний результат:

- Менше збоїв у "важких" файлових системних сценаріях і чіткіша межа між UI та платформним шаром.

## Етап 4 (v1.8). Shell Integration

Ціль: довести Windows-інтеграцію до рівня очікувань від файлового менеджера.

- Перевірити автооновлення каталогів на сценаріях, де зміни приходять від зовнішніх shell-операцій, rename-серій і масових копій.
- Підтягнути та вирівняти поведінку системних асоціацій для складніших edge-case сценаріїв.
- Перевірити й уніфікувати вже наявні модифіковані F-клавіші, persistent console та shell-поведінку для `.bat/.cmd/.ps1` у граничних сценаріях.
- Дотягнути drag-and-drop між панелями та зовнішніми застосунками до більш повного Explorer-like UX.

Очікуваний результат:

- Поведінка ближча до нативної Windows там, де це корисно, але без втрати commander-style UX.

## Етап 5 (v1.9). Quality and Maintainability

Ціль: зменшити ризик регресій і спростити наступні ітерації.

- Додати мінімальні тести для нетривіальної логіки: `FileOperationService`, CSV parsing, path helpers.
- Винести ще частину логіки з `AppForm` і `FileBrowser` у менші сервіси.
- Уніфікувати налаштування, локалізаційні ключі й документувати важливі рішення.
- Поступово прибрати legacy API з уже ізольованого `Dialogs/Legacy/`; фізичну структуру коду за відповідальністю (`Application`, `Browsers`, `Controls`, `Dialogs`, `Editors`, `Viewers`, `Services`, `Infrastructure`) вже впорядковано.
- Налаштувати базову CI-перевірку на `dotnet build`.

Очікуваний результат:

- Менше ручної перевірки на кожну зміну і швидший рефакторинг без страху щось зламати.

## Після бази (v2.0+)

Має сенс після стабілізації попередніх етапів:

- пошук по файлах і вмісту;
- вкладки для панелей;
- undo/redo для файлових операцій;
- запис/редагування вмісту архівів із транзакційним оновленням (поточна archive panel read-only);
- FTP/SFTP або інші віддалені провайдери;
- плагінна модель або розширення команд;
- робота з NTFS links і reparse points: hard links, symbolic links, junctions та mount points.

## Орієнтовні віхи

- `v1.7` — hardening файлових операцій, unified error handling/diagnostics;
- `v1.8` — long-path support, platform hardening, shell edge cases, richer drag-and-drop;
- `v1.9` — перші тести, CI та подальше спрощення архітектури;
- `v2.0+` — пошук, вкладки, undo/redo, запис до архівів, віддалені провайдери.

## Критерії готовності

Етап вважається завершеним, якщо:

- функція працює в обох панелях;
- гарячі клавіші не ламають наявний UX;
- великі каталоги й довгі операції не блокують інтерфейс;
- помилки показуються зрозуміло й, де потрібно, логуються;
- зміни не порушують локалізацію;
- проєкт збирається через `dotnet build`.
